# Career Path — From Junior to QA Architect

## What does a QA career look like?

**Essence:** Many people think QA has only one path: keep testing -> get bored -> change careers. Completely wrong. QA has a **multi-branch career tree** — like an RPG game: you choose your class, level up, and specialize in the direction that suits you.

**Analogy (Real-world example):** Think of a QA career like a tree. The trunk (Junior -> Senior) is something everyone must go through. But at the branches, you have many directions: the technical branch (SDET, Architect), the management branch (Lead, Manager), and the specialist branch (Performance, Security). Each branch leads to a different peak — no branch is "better" than any other.

### Career Map:

```
                          +-- QA Manager ---- Director of QA
                          |    (people management)
                          |
Junior QA -- QA Engineer -- Senior QA --+
 (0-2 yrs)   (2-4 yrs)    (4-7 yrs)   +-- QA Lead / Tech Lead
                                        |    (technical + people leadership)
                                        |
                                        +-- SDET (Dev Engineer in Test)
                                        |    (70% coding, 30% testing)
                                        |
                                        +-- QA Architect
                                        |    (designing test systems)
                                        |
                                        +-- Specialist:
                                             Performance Engineer
                                             Security Engineer
                                             DevOps Engineer (career switch)
```

::: tip Aha Moment
There is no "right" or "wrong" career path. Some people love coding -> SDET. Some prefer management -> Lead/Manager. Some prefer deep technical work -> Architect. **What matters is knowing what you want — and planning to get there.**
:::

## Skills By Level — In Detail

### Junior QA (0-2 years) — "Building the Foundation"

**Essence:** A Junior is like a builder laying the foundation. It's not glamorous, but **a weak foundation means the house collapses**. During this stage you need to: do it right, do it completely, do it consistently.

| Category | Required Skills | Level |
|---|---|---|
| **Testing** | Test case design, Bug reporting, STLC, Test types | Solid |
| **Tools** | Jira, Postman (basic), Browser DevTools | Daily use |
| **Technical** | Basic SQL (SELECT, JOIN, WHERE), API concepts | Understand |
| **Soft skills** | Communication, Attention to detail, Time management | Developing |
| **Mindset** | Learn fast, Ask questions, Be proactive | Essential |

**Daily goals:**
- Write test cases clearly -> Devs can understand them without asking for clarification
- Report bugs completely: steps, actual, expected, evidence -> Dev can reproduce immediately
- Participate in Sprint ceremonies -> contribute your opinion (even small ones)
- Ask "Why?" at least 3 times per day

**Salary range (Vietnam 2024-2025):**

| Company type | Monthly salary range |
|---|---|
| Small outsource | 8-12 million VND |
| Large outsource (FPT, KMS, NAB) | 10-15 million VND |
| Product company (startup) | 10-18 million VND |
| Product company (large/foreign) | 12-20 million VND |

### QA Engineer (2-4 years) — "Starting to Use Power Tools"

**Essence:** Moving from hand-building (manual) to using machines (automation). You don't just test — you start **thinking about strategy**, writing automation, and mentoring juniors.

| Category | Required Skills | Level |
|---|---|---|
| **Automation** | Playwright/Cypress, Page Object Model, CI/CD basic | Hands-on |
| **API Testing** | Postman advanced, API automation, REST concepts | Proficient |
| **Technical** | JavaScript/TypeScript, SQL intermediate, Git workflow | Working knowledge |
| **Testing** | Test strategy thinking, Risk-based testing, Exploratory | Applied |
| **Soft skills** | Mentor juniors, Cross-team communication, Estimation | Growing |

**Goals:**
- Build and maintain an automation test suite (50+ test cases)
- Proficient in API testing — both manual and automated
- Contribute to test strategy decisions (not just execute)
- Mentor 1-2 junior QAs
- Start estimating testing effort accurately

**Salary range (Vietnam 2024-2025):**

| Company type | Monthly salary range |
|---|---|
| Small-medium outsource | 15-22 million VND |
| Large outsource | 18-30 million VND |
| Product company (startup) | 18-32 million VND |
| Product company (large/foreign) | 22-40 million VND |

### Senior QA (4-7 years) — "Quality Architect"

**Essence:** A Senior is not simply "a QA Engineer with more years". A Senior is someone who **sees the big picture**, makes architectural decisions, and influences the entire team. It is like going from a builder to an architect — you no longer lay bricks; you **design the building**.

| Category | Required Skills | Level |
|---|---|---|
| **Architecture** | Automation framework design, Tool selection & evaluation | Ownership |
| **Strategy** | Test strategy for the full product, Risk assessment & mitigation | Leading |
| **Technical** | Performance testing, Security basics, Docker/K8s, Cloud (AWS/GCP) | Practical |
| **Leadership** | Lead testing for a project/team, Process improvement | Driving |
| **Soft skills** | Stakeholder management, Presenting to leadership, Mentoring team | Advanced |

**Goals:**
- Design an automation framework from scratch
- Own the test strategy for the product/team — without being told
- Lead quality initiatives, improve processes
- Influence team decisions (tools, processes, standards)
- Present quality reports to management/stakeholders

**Salary range (Vietnam 2024-2025):**

| Company type | Monthly salary range |
|---|---|
| Large outsource | 25-40 million VND |
| Product company | 30-50 million VND |
| Product company (foreign/big tech) | 40-65 million VND |

### Specialized Paths — Beyond Senior

#### SDET (Software Development Engineer in Test)

**What is an SDET?** An SDET is a **developer who specializes in writing code for testing**. Not a QA who knows code — but a **developer who deeply understands testing**.

```
SDET vs QA Engineer:
- SDET: 70% coding, 30% testing -> Build tools, frameworks, infrastructure
- QA:   30% coding, 70% testing -> Design tests, execute, analyze

What an SDET typically does:
- Build automation frameworks from scratch
- Develop test infrastructure (CI/CD pipelines, test environments)
- Write internal tools (test data generators, mock servers, report dashboards)
- Review production code (not just test code)
- Performance engineering, chaos testing
```

**Salary (VN):** 30-55 million VND/month | Remote: 60-120+ million VND

#### QA Lead / QA Manager

```
QA Lead (Technical + People):
- Manage 3-8 QA engineers
- Sprint/Release planning for QA team
- Quality metrics, reporting, dashboards
- Hiring, training, performance reviews
- Process improvement, tool standardization

QA Manager (People + Strategy):
- Manage multiple teams/projects
- Budget planning (tools, training, hiring)
- Organization-level quality strategy
- Stakeholder management (C-level reporting)
- Build QA culture across the organization
```

**Salary (VN):** 35-60 million VND/month | Remote: 50-100+ million VND

#### QA Architect

```
QA Architect (Visionary):
- Define testing architecture for the entire organization
- Tool selection, evaluation, and standardization across teams
- Cross-team automation strategy
- Performance/Security testing strategy
- Evaluate and adopt new technologies (AI testing, etc.)
- Set quality standards and best practices
```

**Salary (VN):** 40-70 million VND/month | Remote: 70-150+ million VND

## Practical Learning Roadmap — First 12 Months

### Months 1-3: Fundamentals — "Learn to walk before you run"

```
Weeks 1-2: Testing Fundamentals
  +-- ISTQB concepts (test levels, test types, test techniques)
  +-- STLC (Software Testing Life Cycle)
  +-- Understand Agile/Scrum basics

Weeks 3-4: Test Case Design
  +-- Equivalence Partitioning, Boundary Value Analysis
  +-- Decision Table, State Transition
  +-- Practice: Write 50 test cases for a real feature

Weeks 5-8: Bug Reporting + Tools
  +-- Bug lifecycle: New -> Open -> Fixed -> Retest -> Closed
  +-- Jira workflow (create, assign, transition, filter)
  +-- Browser DevTools (Console, Network, Elements)
  +-- Practice: Log 20 bugs for a real website

Weeks 9-12: SQL + API Basics
  +-- SQL: SELECT, WHERE, JOIN, GROUP BY
  +-- Postman: GET, POST, PUT, DELETE
  +-- API concepts: REST, status codes, headers, auth
  +-- Practice: Test 10 API endpoints using Postman
```

### Months 4-6: Tools + API Deep Dive — "Master your weapons"

```
Weeks 1-4: Postman Advanced
  +-- Collections, Environments, Variables
  +-- Pre-request scripts, Tests
  +-- Newman (CLI runner)
  +-- Practice: Complete API test suite for 1 project

Weeks 5-8: SQL Intermediate + Developer Skills
  +-- SQL: Subqueries, Indexes, Transactions
  +-- Git basics: clone, branch, commit, push, pull, merge
  +-- Command line basics
  +-- Practice: Query a production-like database to verify test results

Weeks 9-12: JavaScript Fundamentals
  +-- Variables, functions, arrays, objects
  +-- Async/await, promises
  +-- Node.js basics
  +-- Practice: Write 10 small scripts (file processing, API calls)
```

### Months 7-12: Automation — "From manual to machine"

```
Weeks 1-4: Playwright Basics
  +-- Setup, first test, locators, assertions
  +-- Page Object Model pattern
  +-- Test configuration, fixtures
  +-- Practice: Automate login + 5 core flows

Weeks 5-8: Playwright Advanced + CI/CD
  +-- API testing with Playwright
  +-- Visual testing, mobile emulation
  +-- GitHub Actions: run tests on every PR
  +-- Practice: Full automation suite (20+ tests) with CI

Weeks 9-12: Portfolio Project
  +-- Complete automation project on GitHub
  +-- README with setup instructions, test reports
  +-- CI/CD pipeline with test results dashboard
  +-- Practice: Present the project — prepare for interviews
```

::: tip Aha Moment
This 12-month plan is a **minimum viable roadmap**. You don't need to master everything — you just need **enough to be productive**, then continue learning on the job. 80% of your practical skills will come from real projects, not from courses.
:::

## Certifications — ISTQB: An Honest Assessment

### ISTQB Foundation Level

| Aspect | Honest assessment |
|---|---|
| **Knowledge** | Good. Gives you shared vocabulary and a thinking framework. Knowing terms like "equivalence partitioning" and "boundary value" lets you communicate more professionally with your team. |
| **Resume** | Valuable, especially when you have < 2 years of experience. Many outsource companies **require** ISTQB. Product companies generally **don't care** as much. |
| **Practicality** | ISTQB knowledge is fairly **theoretical**. You won't use 100% of it in your daily work. But it gives you a solid foundation. |
| **Cost** | ~$250 USD (exam) + $0-100 (self-study/books). Good ROI if you are job hunting. |
| **Recommendation** | Take it within your first 1-2 years. Study for 2-3 weeks, pass on the first attempt. |

### Other Certifications

| Cert | When to take it | Practical value |
|---|---|---|
| **ISTQB Advanced (Test Manager)** | 3+ years, aiming for Lead/Manager | Theory-heavy but structured. Useful for outsource companies. |
| **ISTQB Advanced (Test Automation)** | 3+ years, focused on automation | Fairly outdated compared to industry practice. Optional. |
| **AWS/Azure Cloud Cert** | When your project uses cloud | High value if you test cloud-native apps. Cross-functional skill. |
| **Certified Scrum Master (CSM)** | Want to understand Agile more deeply | Valuable for communication + process improvement. Nice to have. |

**Straight talk:**
- Certifications **help open doors** (pass the HR filter) but **don't make you better at the job**
- 1 automation project on GitHub is **worth more** than 3 certificates in an interview
- If you must choose: **project portfolio > certification > course completion**
- Exception: Outsource companies (FPT, KMS, TMA...) often require ISTQB -> take it

## Interview Questions By Level

### Junior — "Do you understand the fundamentals?"

| Question | Expected Answer Depth |
|---|---|
| "Write test cases for Login?" | 10-15 TCs: positive (valid credentials), negative (wrong password, empty fields, SQL injection, locked account), boundary (max length), UX (remember me, forgot password link) |
| "What is the bug lifecycle?" | New -> Assigned -> Open -> Fixed -> Retest -> Closed/Reopened. Explain each stage + who is responsible. Bonus: Deferred, Duplicate, Not a Bug |
| "Smoke test vs Sanity test?" | Smoke = broad and shallow (test many features, a little of each — "is the house on fire?"). Sanity = narrow and deep (test 1 specific feature thoroughly — "does this work correctly?"). Give a real-world example. |
| "Severity vs Priority?" | Severity = how serious the issue is technically. Priority = how urgently it needs fixing per business needs. Example: Wrong logo on the homepage = Low severity, High priority (affects brand) |
| "What do you test for a search box?" | Functional (correct results, incorrect, empty, special chars), UI (placeholder, clear button, auto-suggest), Performance (response time), Security (XSS, SQL injection), Usability (keyboard Enter, mobile) |

### Mid-level — "How do you think?"

| Question | Expected Answer Depth |
|---|---|
| "Design a test strategy for feature X?" | Analyze risks -> Choose test approach (manual vs auto) -> Define scope -> Prioritize (critical paths first) -> Resource planning -> Entry/Exit criteria. Don't just list TCs — explain WHY. |
| "When to automate, when to stay manual?" | ROI analysis: automate when stable + repeated + high-risk. Manual when: exploratory, one-time, UI changing frequently. Decision matrix + real examples from past projects. |
| "Explain the POM pattern?" | Page Object Model: each page = 1 class. Separate locators + actions from tests. Benefits: maintainable, reusable, readable. Bonus: draw a diagram, provide a code example. |
| "How do you handle flaky tests?" | Root causes: timing issues, test data dependencies, environment instability. Strategies: explicit waits (no hard-coded sleep), independent test data, retry mechanism, quarantine flaky tests. Show real experience. |

### Senior — "How do you lead and make decisions?"

| Question | Expected Answer Depth |
|---|---|
| "Build an automation framework from scratch?" | Architecture decisions: language choice (why TS not Java?), framework (why Playwright?), patterns (POM, fixtures, custom reporters), CI/CD integration, parallel execution, reporting strategy, maintenance plan. Discuss tradeoffs. |
| "Improve the quality process for a team?" | Current state analysis (metrics: defect leakage, automation coverage, cycle time) -> Identify gaps -> Propose improvements (shift-left, automation roadmap, code review for tests) -> Measure impact -> Iterate. Show a metrics-driven approach. |
| "What quality metrics do you track?" | Test coverage (requirement + code), Pass rate (per sprint), Defect leakage rate (escaped bugs), Automation ROI, Mean time to detect/fix, Flaky test rate. **Important: explain what each metric TELLS YOU about quality.** |
| "Build a QA team of 5 people?" | Hiring criteria, onboarding plan, skills matrix, mentoring program, knowledge sharing (weekly sessions), tool standardization, career growth plan for each member. Show leadership thinking. |

## Portfolio Building — Your GitHub Profile Is Your Best Resume

### Ideal GitHub Portfolio structure:

```
github.com/your-name/
+-- playwright-automation-project/     <- Main showcase
|   +-- tests/
|   |   +-- e2e/        (10+ end-to-end tests)
|   |   +-- api/        (10+ API tests)
|   |   +-- visual/     (visual regression tests)
|   +-- pages/           (Page Object Model)
|   +-- fixtures/        (test data, custom fixtures)
|   +-- .github/workflows/  (CI/CD pipeline)
|   +-- playwright.config.ts
|   +-- package.json
|   +-- README.md        (setup, run, results, architecture diagram)
|
+-- api-test-collection/               <- API testing skills
|   +-- postman-collections/
|   +-- automated-api-tests/
|   +-- README.md
|
+-- performance-tests/                 <- Bonus: shows depth
|   +-- k6-scripts/
|   +-- README.md
|
+-- qa-learning-notes/                 <- Shows continuous learning
    +-- test-strategy-templates/
    +-- bug-report-templates/
    +-- cheat-sheets/
```

### Tips for your GitHub Profile:

```
1. High-quality README.md:
   - Clear project description
   - Setup instructions (anyone who clones it can run it)
   - Screenshot/GIF of tests running
   - Architecture diagram
   - Link to test results/reports

2. Working CI/CD pipeline:
   - GitHub Actions badge (green) on the README
   - Tests run automatically on every push
   - Test report published (GitHub Pages or Allure)

3. Code quality:
   - Consistent naming conventions
   - Comments explain WHY (not WHAT)
   - Clean folder structure
   - No hardcoded values -> use config/env

4. Activity:
   - Commit regularly (at least 3-4 times/week)
   - Green contribution graph -> shows consistency
   - Star/Fork related projects -> shows interest
```

::: tip Aha Moment
Interviewers are more impressed by **real working projects** than by any certificate. When you open GitHub and show: "Here is an automation framework I built, CI/CD runs automatically, 95% pass rate, 80% code coverage" — you have already won 80% of the interview. **Show, don't tell.**
:::

## Final Words — The Truth About a QA Career

```
A QA career has no shortcuts.

There is no "become Senior in 3 months".
There is no "learn 1 tool and you're set".
There is no "certification replaces experience".

But if you:
  - Learn consistently every day (15-30 minutes — consistency > intensity)
  - Practice on real projects (don't just watch tutorials)
  - Build your portfolio on GitHub (show, don't tell)
  - Train your mindset + communication (re-read the previous 2 articles)
  - Join the community (ask, share, network)

-> You will progress faster than 90% of others.

This knowledge base is the first step.
The next step is YOURS — turn knowledge into action.
```

::: tip Final Aha Moment
Don't compare yourself to others. Compare yourself today to yourself last week. If this week you learned 1 more SQL query, wrote 1 more automation test, or understood 1 more concept — you are on the right track. **Progress > Perfection.**
:::
