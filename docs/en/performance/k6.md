# K6

## The Essence: K6 = The "Developer-Friendly" Version of JMeter

If JMeter is a **classic DSLR camera** (GUI, lots of buttons, anyone can use it), then K6 is a **mirrorless camera** (compact, powerful, requires some know-how).

> **K6** = a load testing tool written in **JavaScript**. Lightweight, fast, and **extremely easy** to integrate with CI/CD. Developed by Grafana Labs.

### Why K6?

| Criteria | K6 | JMeter |
|---|---|---|
| **Scripting** | JavaScript (familiar to QA automation engineers) | GUI-based (XML) |
| **CI/CD** | Native support -- just one command | Requires plugins |
| **Performance** | Very lightweight, low RAM usage | Heavy (Java-based) |
| **Code review** | Scripts live in Git, human-readable | JMX files are not readable |
| **Learning curve** | Easy if you know JS | Easy if you prefer GUI |

::: tip Aha moment
K6 scripts are plain JavaScript -- you already learned JS for Playwright, and now you can use THE SAME language to write performance tests. No need to learn a new tool!
:::

## Your First Test -- Every Line Explained

```javascript
// load-test.js
import http from 'k6/http';
// ^^^ Import K6's HTTP module -- used to send requests
import { check, sleep } from 'k6';
// ^^^ check = verify the response is correct
//     sleep = wait (simulates real users pausing between actions)

// === CONFIGURATION: how many users, how long to run ===
export const options = {
  vus: 10,            // 10 virtual users
  duration: '30s',    // run for 30 seconds
};

// === TEST FUNCTION: each virtual user runs this function REPEATEDLY ===
export default function () {
  // Send a GET request to the server
  const res = http.get('https://api.example.com/products');

  // Verify the response
  check(res, {
    'status is 200': (r) => r.status === 200,
    // ^^^ Response status must be 200 (OK)

    'response faster than 500ms': (r) => r.timings.duration < 500,
    // ^^^ Response time must be < 500ms

    'data is returned': (r) => r.json().length > 0,
    // ^^^ Body must contain data (not empty)
  });

  sleep(1);
  // ^^^ Wait 1 second before repeating
  // (simulates real users: read the page, then click again)
}
```

```bash
# Run the test
k6 run load-test.js

# K6 will display results:
#   http_req_duration...: avg=234ms  p(90)=450ms  p(95)=680ms
#   http_req_failed.....: 0.00%
#   vus.................: 10
#   iterations..........: 280
```

## Load Patterns -- Simulating Different Types of Load

### Constant Load = "Keep the same number of customers"

```javascript
// Simple: 50 users running for 5 minutes
export const options = {
  vus: 50,           // 50 virtual users
  duration: '5m',    // run for 5 minutes
};
```

### Ramp-up / Ramp-down = "Gradually increase, gradually decrease" (most realistic)

```javascript
export const options = {
  stages: [
    // stages = different "phases" -- like a song with multiple sections
    { duration: '2m', target: 50 },   // First 2 minutes: ramp up from 0 to 50 users
    { duration: '5m', target: 50 },   // Next 5 minutes: hold at 50 users (measure stable load)
    { duration: '2m', target: 100 },  // 2 minutes: ramp up to 100 users
    { duration: '5m', target: 100 },  // 5 minutes: hold at 100 users
    { duration: '2m', target: 0 },    // Final 2 minutes: ramp down to 0 (end)
  ],
};
```

### Spike Test = "Sudden surge"

```javascript
export const options = {
  stages: [
    { duration: '1m', target: 50 },    // Normal load
    { duration: '10s', target: 500 },   // SPIKE! jump from 50 to 500 in 10 seconds
    { duration: '2m', target: 500 },    // Hold high load for 2 minutes
    { duration: '10s', target: 50 },    // Drop back to normal
    { duration: '2m', target: 50 },     // Hold normal load -- does the system recover?
  ],
};
```

## Thresholds = Automatic "PASS/FAIL" Criteria

Thresholds are like declaring upfront: "If P95 > 2 seconds or error > 1%, then it is a FAIL." K6 will **automatically** evaluate PASS/FAIL for you.

```javascript
export const options = {
  stages: [
    { duration: '2m', target: 100 },
    { duration: '5m', target: 100 },
    { duration: '1m', target: 0 },
  ],

  thresholds: {
    // === PASS/FAIL CRITERIA ===

    http_req_duration: [
      'p(90) < 1000',   // 90% of requests must be faster than 1 second
      'p(95) < 2000',   // 95% of requests must be faster than 2 seconds
    ],

    http_req_failed: [
      'rate < 0.01',    // Error rate must be < 1%
    ],

    // Threshold specific to an individual endpoint
    'http_req_duration{name:login}': [
      'p(95) < 1500',   // Login API must be faster than 1.5 seconds
    ],
  },
};
```

### Result when PASS:

```
v  http_req_duration........: avg=450ms  p(90)=890ms  p(95)=1200ms
     v { name:login }.......: avg=380ms  p(95)=980ms
v  http_req_failed...........: 0.5%  v < 1%
```

### Result when FAIL:

```
x  http_req_duration........: avg=3200ms  p(90)=5000ms  p(95)=8000ms
     x p(95) < 2000
# K6 exit code = 99 (FAIL) --> CI/CD pipeline will FAIL
# --> Deploy is BLOCKED automatically!
```

::: tip Aha moment
Thresholds + CI/CD = **automatically block deploys** when performance does not meet criteria. No need for a QA engineer to sit and review results manually -- K6 decides for you.
:::

## Practical Example: E-commerce Load Test

```javascript
// ecommerce-load-test.js
import http from 'k6/http';
import { check, group, sleep } from 'k6';
// ^^^ group = group requests into a named scenario (easier to read reports)

const BASE_URL = 'https://api.example.com';

// === CONFIGURATION ===
export const options = {
  stages: [
    { duration: '2m', target: 50 },   // Ramp up to 50 users
    { duration: '5m', target: 50 },   // Hold 50 users for 5 minutes
    { duration: '1m', target: 0 },    // Ramp down to 0
  ],
  thresholds: {
    http_req_duration: ['p(95) < 2000'],  // 95% of requests < 2 seconds
    http_req_failed: ['rate < 0.01'],      // Error < 1%
  },
};

// === SCENARIO: User browses products and adds to cart ===
export default function () {

  // Step 1: View product list
  group('View product list', () => {
    const res = http.get(`${BASE_URL}/api/products?page=1&limit=20`);
    check(res, {
      'product list returns 200': (r) => r.status === 200,
      'products exist': (r) => r.json().data.length > 0,
    });
    sleep(2);
    // ^^^ User reads the list for 2 seconds before clicking a product
  });

  // Step 2: View product details
  group('View product details', () => {
    const res = http.get(`${BASE_URL}/api/products/1`);
    check(res, {
      'details return 200': (r) => r.status === 200,
      'product has a name': (r) => r.json().name !== '',
    });
    sleep(1);
    // ^^^ Read the details for 1 second
  });

  // Step 3: Add to cart
  group('Add to cart', () => {
    const payload = JSON.stringify({
      productId: 1,
      quantity: 1,
    });
    const headers = { 'Content-Type': 'application/json' };

    const res = http.post(`${BASE_URL}/api/cart`, payload, { headers });
    check(res, {
      'added to cart successfully': (r) => r.status === 200,
    });
    sleep(1);
  });
}
```

## CI/CD Integration -- Running K6 Automatically in GitHub Actions

```yaml
# .github/workflows/performance.yml
name: Performance Tests

on:
  schedule:
    - cron: '0 2 * * 1'
    # ^^^ Every Monday at 2AM UTC (9AM Vietnam time)
    # Run weekly to track performance over time

jobs:
  k6-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Use the ready-made K6 action -- no installation needed
      - uses: grafana/k6-action@v0.3.1
        with:
          filename: tests/performance/load-test.js
          # ^^^ Path to the K6 test file
        env:
          BASE_URL: https://staging.example.com
          # ^^^ Run on staging, NOT on production!

      # Save results for later review
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: k6-results
          path: summary.json
```

::: tip Aha moment
K6 + GitHub Actions = **automated weekly performance testing**. If this week P95 jumps from 800ms to 2000ms, you know RIGHT AWAY -- instead of waiting for users to complain.
:::

## Chapter Summary

| Concept | Analogy | How to use in K6 |
|---|---|---|
| **VUs** | Virtual users | `options.vus` or `stages` |
| **Duration** | How long to run | `options.duration` |
| **Stages** | Phases (ramp up, hold, ramp down) | Ramp-up, sustain, ramp-down |
| **Checks** | Verify response | `check(res, { ... })` |
| **Thresholds** | PASS/FAIL criteria | `options.thresholds` -- auto-fail CI/CD |
| **Groups** | Group requests | `group('name', () => { ... })` |
| **Sleep** | User pauses | `sleep(1)` -- realistic simulation |
| **CI/CD** | Automated runs | GitHub Actions + grafana/k6-action |
